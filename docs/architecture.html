▸ Emulator Layer

Single Emulator: pygba (mGBA)

Handles both GB and GBA from one code path. GB games auto-cropped to remove Super Game Boy borders.

GamePlatformRawAfter Crop
Pokémon BlueGB256×224144×160
Pokémon RedGB256×224144×160
Pokémon LeafGreenGBA240×160— (no crop)

GB Crop Strategy

SGB border = decorative noise for vision model. Crop to game-only region:

if rom_is_gb:
    screen = screen[40:184, 48:208]  # → 144×160

▸ Model Architecture

Vision Model: google/gemma-3-12b-it

Multimodal. Reads raw screenshots. Outputs structured JSON — never narration.

Cost$0.05/M in + $0.15/M out
Context131K tokens
Per screenshot~$0.0001
What it extractsscreen_type, enemy_name, hp%, text_lines, menu_options

Thinking Model: openrouter/owl-alpha

Text-only. Receives assembled prompt + vision output. Returns tool calls.

CostFREE ($0.00)
Context1M tokens
Per decision$0.00000
What it doesSelects next action from tool schema

Why two models? Vision models are expensive for text reasoning. Text models are cheap/free for decision-making. Gemma reads the screen once; owl-alpha makes decisions all day for free. Total cost per gameplay hour: ~$0.04.

▸ Tool Schema (Thinking Model → Emulator)

press_button

{
  "button": "A|B|UP|DOWN|
             LEFT|RIGHT|
             START|SELECT",
  "duration": 5
}

Press and hold for N frames, then release

wait

{
  "frames": 60
}

Advance emulator without pressing anything (1 frame = 1/60s)

combo

{
  "buttons": ["UP","A"],
  "duration": 5
}

Press multiple buttons simultaneously

▸ Stacked Prompt System

Prompts are layered configs selected by game generation × screen type. The vision model's output determines which stack loads.

Config Structure

configs/prompts/
├── gen1/              # Red/Blue/Yellow
│   ├── battle.yaml
│   ├── overworld.yaml
│   ├── menu.yaml
│   └── dialog.yaml
└── gen3/              # FireRed/LeafGreen
    ├── battle.yaml
    ├── overworld.yaml
    ├── menu.yaml
    └── dialog.yaml

Each YAML Contains

system_prompt — Role, rules, what game this is, what you control
tool_definitions — Available commands for this screen type
observation_template — Injects vision output: "Enemy: {name} HP:{hp}%"
memory_context — Recent actions, active goals, party status
few_shot_examples — 2-3 examples of correct tool calls for this screen

▸ Vision Prompt Design

The vision prompt is the most critical piece. It must extract specific actionable data — not describe the scene.

Questions It Must Answer

  • • What screen type? (battle / overworld / menu / dialog / title)
  • • If battle: enemy Pokémon name? Your HP bar %? Enemy HP bar %?
  • • What moves are shown? Any status icons?
  • • If overworld: adjacent tiles? NPCs? Signs? Grass? Exits?
  • • If menu: which menu? What's highlighted? How many options?
  • • If dialog: exact text shown? Is there a yes/no prompt?
  • • Any text boxes? What do they say — top line verbatim?

Output Format (JSON)

{
  "screen_type": "battle",
  "enemy_pokemon": "Pidgey",
  "player_hp_pct": 72,
  "enemy_hp_pct": 34,
  "text_lines": [
    "What will PIDGEY do?",
    "FIGHT  BAG",
    "POKéMON  RUN"
  ],
  "menu_items": ["FIGHT","BAG","POKéMON","RUN"],
  "adjacent_tiles": [],
  "status_icons": [],
  "dialog_prompt": null
}

⚠️ System prompt must forbid narration. No "I can see a Pokémon battle..." — JSON only.

▸ Decision Loop (Per Tick)

1. Capture — pygba renders frame, crop if GB

2. Vision — Gemma 12B reads screenshot → structured JSON

3. Select — screen_type + generation → choose prompt stack

4. Assemble — system + tools + vision data + memory + examples → final prompt

5. Decide — owl-alpha returns tool call JSON

6. Execute — tool call → emulator button press

7. Repeat — advance N frames, go to 1

State Machine

[BOOT] → [TITLE] → [NEW GAME]

    ▼

    └→ [SHOP]

▸ Build Phases

1

Emulator Wrapper + Tool Executor

Replace old emulator.py with unified pygba wrapper. GB crop, tool schema, press_button/wait/combo execution. Verify screenshots work for Blue + LeafGreen.

2

Prompt Config System

YAML configs for gen1/gen3 × battle/overworld/menu/dialog. Prompt assembler that layers system → tools → observation → memory → examples. Few-shot examples for each screen type.

3

Vision Pipeline

Wire Gemma 3 12B via OpenRouter. Screenshot → base64 → structured JSON. Retry on parse failure. Vision output cached per screenshot.

4

Decision Pipeline

Vision output → prompt selection → assembly → owl-alpha → tool call parse → execute. Memory injection (last 10 actions, current goal, party status). Fallback on parse failures (default "press A").

5

Autonomous Cron Loop

coding-hermes cron job. Each tick: run N frames, commit checkpoint, push results. Database logging for every decision. Replay support for debugging.
